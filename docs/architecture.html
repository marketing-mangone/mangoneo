Sección 01

Visión y Alcance del Proyecto

Una plataforma interna centralizada que consolide métricas, documentación, assets de marca, KPIs y recursos operativos del departamento de marketing en un solo punto de acceso.

Problema que Resuelve

Actualmente la información del departamento vive dispersa entre Google Drive, Dropbox, HubSpot, Monday.com, Canva y múltiples herramientas. El Hub elimina la fragmentación y crea una fuente única de verdad para el equipo de 6 personas, con capacidad de escalar a medida que la firma crece.

Objetivos Técnicos

🎯

Centralización

Un solo punto de entrada para métricas, manuales, SOPs, JDs y recursos del departamento. Sin necesidad de navegar entre 8+ herramientas.

📊

Visibilidad en Tiempo Real

Dashboards con KPIs actualizados automáticamente desde HubSpot, Google Analytics, Meta y otras fuentes via API.

🔒

Seguridad

Control de acceso basado en roles. Los documentos internos, métricas financieras y datos sensibles del equipo protegidos con autenticación robusta.

⚡

Escalabilidad

Arquitectura modular que permite agregar nuevos módulos (presupuesto, campaign tracker, asset library) sin refactorizar.

Sección 02

Stack Tecnológico

Selección de tecnologías optimizada para un equipo pequeño con capacidad de crecimiento, priorizando estabilidad, ecosistema maduro y velocidad de desarrollo.

Backend

Django 5.x Django REST Framework Python 3.12+ Celery + Redis django-storages

Django es la elección natural: ORM potente, admin panel integrado, sistema de autenticación robusto, y un ecosistema de paquetes maduro. Django REST Framework expone la API que consume el frontend. Celery gestiona tareas asíncronas como sincronización de métricas, generación de reportes y procesamiento de archivos.

Frontend

Next.js 14+ React 18+ TypeScript Tailwind CSS shadcn/ui Recharts

Next.js ofrece rendering híbrido (SSR + CSR), routing integrado, y optimización automática. Tailwind + shadcn/ui permiten construir interfaces profesionales con alta velocidad sin sacrificar calidad visual. Recharts para las visualizaciones de datos y dashboards.

Base de Datos

PostgreSQL 16 Redis
¿Por qué PostgreSQL?

PostgreSQL es la base de datos recomendada por Django y la más robusta para este caso de uso. Soporta tipos JSON nativos (útil para métricas flexibles), full-text search en español (para buscar dentro de documentos), campos de array, y excelente manejo de archivos binarios grandes a través de referencias. Redis funciona como cache, message broker para Celery, y almacenamiento de sesiones.

Almacenamiento de Archivos

AWS S3 / Cloudflare R2 django-storages Pillow

Los archivos (PDFs del brand manual, imágenes, SOPs en Word/PDF, videos) no se almacenan en la base de datos. Se suben a object storage (S3 o R2 para mejor costo) y la DB solo guarda la referencia (URL, metadata, permisos). Esto mantiene la DB ligera y permite servir archivos pesados directamente desde CDN.

Comparativa de Alternativas Consideradas

Componente Elegido Alternativa Razón de la decisión
Frontend Next.js Vue/Nuxt, Angular Mayor ecosistema, SSR nativo, mejor con shadcn/ui
DB PostgreSQL MySQL, MongoDB JSON nativo, full-text search, mejor con Django ORM
Storage Cloudflare R2 AWS S3, GCS Sin egress fees, compatible con S3 API, menor costo
Cache Redis Memcached También sirve como broker para Celery
Task Queue Celery Django-Q, Dramatiq Estándar de facto en Django, mejor documentación

class MetricDefinition(models.Model):
    """Definición de cada KPI que trackeamos"""
    name = models.CharField()               # "Leads Generados", "Costo por Lead"
    slug = models.SlugField(unique=True)
    category = models.CharField()            # acquisition, engagement, conversion, brand
    source = models.CharField()              # hubspot, google_analytics, meta, manual
    unit = models.CharField()                # count, currency, percentage, time
    target_value = models.DecimalField(null=True)
    description = models.TextField()
    formula = models.TextField(blank=True)   # Cómo se calcula
    owner = models.ForeignKey(User, null=True, on_delete=models.SET_NULL)

class MetricSnapshot(models.Model):
    """Valores capturados periódicamente"""
    metric = models.ForeignKey(MetricDefinition, related_name='snapshots')
    value = models.DecimalField(max_digits=14, decimal_places=4)
    period_start = models.DateField()
    period_end = models.DateField()
    period_type = models.CharField()         # daily, weekly, monthly
    raw_data = models.JSONField(default=dict)  # Data cruda de la fuente
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        unique_together = ['metric', 'period_start', 'period_type']
        indexes = [
            models.Index(fields=['metric', 'period_start']),
        ]

Sección 06

Seguridad y Autenticación

Múltiples capas de seguridad adaptadas a una firma de abogados que maneja información sensible de clientes y del negocio.

Autenticación

Login (email + password)
→
Django genera JWT
→
Access Token (15min)
+
Refresh Token (7d)

Se recomienda djangorestframework-simplejwt para la gestión de tokens. El access token tiene vida corta (15 minutos) y se renueva automáticamente con el refresh token, que tiene rotación (cada uso genera uno nuevo e invalida el anterior).

MFA Recomendado para Fase 2

Para el lanzamiento inicial, email + password con JWT es suficiente dado que es un sistema interno de 6-10 usuarios. En Fase 2 se puede agregar TOTP (Google Authenticator) o SSO con Google Workspace si la firma ya lo usa.

Capas de Seguridad

🔐
HTTPS Obligatorio

TLS 1.3 en toda la comunicación. HSTS headers activados. Certificados via Let's Encrypt o Cloudflare.

🛡️
CORS Restrictivo

Solo el dominio del frontend permitido. No wildcards. Headers de seguridad configurados.

🚦
Rate Limiting

DRF throttling por usuario y por IP. Protección contra fuerza bruta en login (5 intentos / lockout 15min).

📝
Audit Log

Registro de acciones sensibles: login, descarga de documentos, cambios de permisos, edición de métricas.

🔒
Encriptación en Reposo

PostgreSQL con pgcrypto para campos sensibles. R2/S3 con server-side encryption para archivos.

🧹
Input Validation

Serializers de DRF validan todo input. Protección CSRF, XSS, SQL injection out-of-the-box con Django.

Sección 07

Gestión de Archivos y Media

Estrategia para manejar documentos pesados, imágenes, videos y assets de marca sin degradar performance.

Flujo de Upload

Usuario sube archivo
→
Validación (tipo, tamaño)
→
Presigned URL → R2
→
Celery: thumbnail + metadata
→
DB guarda referencia

Los archivos nunca pasan por el servidor Django en producción. El frontend obtiene un presigned URL de R2/S3 y sube directamente al bucket. Esto reduce carga del servidor y permite archivos grandes (videos, PDFs pesados del brand manual).

Tipos de Archivo Soportados

Categoría Formatos Límite Procesamiento
Documentos PDF, DOCX, PPTX, XLSX, MD 50 MB Extracción de texto para búsqueda, preview thumbnail
Imágenes PNG, JPG, SVG, WebP 25 MB Resize automático, thumbnail 400px, conversión a WebP
Brand Assets AI, EPS, PSD, PDF, SVG, PNG 100 MB Thumbnail del preview, metadata de dimensiones
Video MP4, MOV 500 MB Thumbnail del primer frame, duración

Búsqueda Full-Text dentro de Documentos

Al subir un PDF o DOCX, un task de Celery extrae el texto y lo indexa en PostgreSQL usando SearchVectorField. Esto permite buscar "VAWA" o "Golden Tickets" y encontrar todos los documentos que mencionan esos términos, sin depender de servicios externos como Elasticsearch.

Ejemplo: Celery task para HubSpot sync 
# apps/integrations/tasks.py

@shared_task(bind=True, max_retries=3)
def sync_hubspot_metrics(self):
    """Sincroniza métricas de HubSpot cada 30 min"""
    try:
        client = HubSpotClient(api_key=settings.HUBSPOT_API_KEY)
        
        # Leads generados hoy
        leads_today = client.crm.contacts.search(
            filter_groups=[{
                "filters": [{
                    "propertyName": "createdate",
                    "operator": "GTE",
                    "value": today_start_ms()
                }]
            }]
        )
        
        MetricSnapshot.objects.update_or_create(
            metric=MetricDefinition.objects.get(slug='leads-diarios'),
            period_start=date.today(),
            period_type='daily',
            defaults={
                'value': leads_today.total,
                'period_end': date.today(),
                'raw_data': {'source': 'hubspot', 'total': leads_today.total}
            }
        )
    except Exception as exc:
        self.retry(exc=exc, countdown=60 * 5)
Sección 09

Infraestructura y Deploy

Setup de producción containerizado, diseñado para ser mantenible por un equipo pequeño con costo controlado.

Opción Recomendada: Railway / Render

Para un equipo de 6-10 usuarios internos, un PaaS manejado elimina la complejidad de administrar servidores propios. Railway o Render permiten deploy desde Git con zero-config para Django + PostgreSQL + Redis.

🚀

Railway (Recomendado)

Deploy de Django + PostgreSQL + Redis en un click. $5/mes por servicio, incluye SSL automático, custom domains, y logs. Ideal para internal tools.

▲

Vercel (Frontend)

Next.js en Vercel es deploy nativo. Free tier suficiente para un Hub interno. Edge functions, ISR, image optimization incluidos.

☁️

Cloudflare R2 (Storage)

$0.015/GB almacenado, sin costo de egress (lectura gratuita). Para el volumen de archivos del departamento, costo < $5/mes.

Sección 12

Decisiones Técnicas Clave

Registro de decisiones arquitectónicas (ADRs) que fundamentan las elecciones del proyecto.

ADR-001: API desacoplada vs. Django Templates

Decisión: API REST (DRF) + Next.js separado.

Razón: Aunque Django templates es más rápido de desarrollar inicialmente, un frontend desacoplado permite: interfaces más ricas y reactivas (drag-and-drop, dashboards en tiempo real, transiciones), que Andrés (frontend/SEO) y un backend developer trabajen en paralelo, y eventual uso de la misma API para una app móvil o integraciones futuras.

ADR-002: PostgreSQL full-text search vs. Elasticsearch

Decisión: PostgreSQL nativo con SearchVectorField + SearchQuery.

Razón: Para un catálogo de ~100-500 documentos, PostgreSQL maneja full-text search sin problema y en español con la configuración spanish. Elasticsearch agregaría un servicio adicional a mantener, costo de infra, y complejidad operacional que no se justifica para el volumen actual. Se puede migrar a Elasticsearch si el catálogo supera los 10,000 documentos.

ADR-003: Presigned URLs vs. Upload through Django

Decisión: Upload directo a R2 via presigned URLs.

Razón: Los archivos (especialmente videos de Gloriana y assets de Sara) pueden ser pesados. Pasar por Django significa que el servidor debe buffearear todo el archivo en memoria, lo cual limita concurrencia y puede causar timeouts. Con presigned URLs, el browser sube directamente a R2, Django solo genera el URL firmado y registra la metadata post-upload.

ADR-004: Monorepo vs. Repos separados

Decisión: Monorepo con carpetas /backend y /frontend.

Razón: Con un equipo tan pequeño (1-2 developers), un monorepo simplifica: un solo PR para features full-stack, docker-compose levanta todo, y no hay necesidad de gestionar versionamiento de API entre repos. Cuando el equipo crezca, se puede dividir.

ADR-005: Métricas como snapshots vs. real-time proxy

Decisión: Celery sincroniza y almacena snapshots en PostgreSQL.

Razón: Hacer proxy directo a las APIs en cada page load significaría latencia alta y riesgo de rate limits. Almacenando snapshots periódicos, el dashboard carga instantáneamente y tenemos historial para tendencias. El tradeoff (datos con delay de hasta 30 min) es aceptable para un dashboard de marketing interno.
